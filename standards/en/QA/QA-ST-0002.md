# QA-ST-0002 ‚Äî Quality Standard for DJIN Members (QA) ‚Äî Operational Version

This document defines **the minimum requirements** for any client request to advance to Architecture.

It is applied by **DJIN Members (QA)** at **QA[1] (intake)** and **QA[2] (final validation)**.

The absence of any listed requirement implies **blocking**, returning to the client, and requesting clarification.

## üë§ Who Executes This Standard

This standard is applied by **DJIN Members** responsible for quality triage (QA).

Any DJIN Member in a QA role must follow these guidelines when evaluating client requests.

---

# 1. FEATURE / ADJUSTMENT / FEATURE REMOVAL

To be considered valid, the request must contain:

1. **Clear purpose** ‚Äì what objective the client wants to achieve.
2. **Problem being solved** ‚Äì what pain point or failure the change resolves.
3. **System location** ‚Äì screen, functionality, or affected flow.
4. **Illustrative images** (when applicable).
5. **Flowchart of the desired flow** (when applicable).
6. **Who will use it** and **real-world usage example**.

### Standard message to client when information is missing

```
Thank you for contacting us.
We apologize, but we were unable to clearly confirm the purpose or the issue this request aims to resolve, so we would appreciate if you could provide additional explanation.
Additionally, it would be very helpful if you could share which screen or functionality this request pertains to, along with images or specific examples.
We apologize for the inconvenience and thank you for your cooperation.
```

# 3. BUG REPORT

Minimum requirements:

1. **Where it occurred** (screen, URL, system).
2. **When it occurred** (date and time).
3. **Who** (name, email, ID).
4. **What was expected**.
5. **What happened** (detailed; images/video help and prevent rework).

### Standard message

```

To accurately understand the situation, could you please provide the following information:
„ÉªScreen or URL where it occurred
„ÉªDate and time of occurrence
„ÉªName / ID / email address of the affected person
```

# 6. ALIGNMENT REQUEST

Possible cases:

### (i) Missing deadline

```
Thank you for contacting us.
Just to confirm,
what is the expected delivery date (deadline) for this matter?
We would appreciate if you could inform us for accurate schedule coordination.
```

### (ii) Missing priority

```
Thank you for your request.
Regarding the priority of this matter, could you please inform us
of your desired level, such as [Priority 1-3] or "Urgent"?
We will adjust the internal response order according to priority.
```

### (iii) Confirm understanding

```
Thank you for contacting us.
To respond accurately, please let us confirm
whether our understanding of the intent and request content is correct.
(‚Äª Briefly bullet-point what we understand)
Could you please confirm if the above is correct?

```

### (iv) Direction change

```

Just to confirm,
is this an instruction to "temporarily suspend current work and prioritize different work"?
As this affects the overall project, we would appreciate your confirmation.
Thank you for your cooperation.
```
