# QA-ST-0002 — DJIN Members (QA) のための品質基準 — 運用版

本文書は、クライアントからのリクエストがアーキテクチャに進むための**最低要件**を定義します。

これは**DJIN Members (QA)** によって**QA[1]（受付）**と**QA[2]（最終検証）**で適用されます。

記載されている要件のいずれかが欠けている場合、**ブロック**し、クライアントに戻して説明を求めます。

---

## 👤 この基準を実行する人

この基準は、品質トリアージ（QA）を担当する**DJIN Members**によって適用されます。

QA役割のすべてのDJIN Memberは、クライアントリクエストを評価する際にこれらのガイドラインに従う必要があります。

---

# 1. 機能追加 / 調整 / 機能削除

有効とみなされるためには、リクエストに以下が含まれている必要があります：

1. **明確な目的** – クライアントが達成したい目標。
2. **解決される問題** – 変更が解決する課題または不具合。
3. **システム内の場所** – 影響を受ける画面、機能、またはフロー。
4. **説明画像**（該当する場合）。
5. **希望するフローのフローチャート**（該当する場合）。
6. **誰が使用するか**と**実際の使用例**。

### 情報が不足している場合のクライアントへの標準メッセージ

```
ご連絡ありがとうございます。
大変恐れ入りますが、本件の目的や解決したい課題が明確に確認できなかったため、追加のご説明をお願いできますでしょうか。
また、どの画面・どの機能に対するご依頼か、併せて画像や具体例などをご共有いただけますと大変助かります。
お手数をおかけいたしますが、よろしくお願い申し上げます。
```

---

# 2. データリクエスト

最低要件：

1. **データの形式**（列、フォーマット、可能であれば例）。
2. **環境**（青 / 緑）。
3. **フィルター** – 時間範囲、条件、制限など。

### 標準メッセージ

```
ご依頼いただきありがとうございます。
必要な情報が一部不足しているため、以下についてご教示いただけますでしょうか。
・必要なデータ項目（カラム）
・対象の環境（青／緑）
・範囲および適応条件など
お忙しいところ恐縮ですが、ご確認のほどよろしくお願いいたします。
```

---

# 3. バグレポート

最低要件：

1. **発生場所**（画面、URL、システム）。
2. **発生時刻**（日付と時刻）。
3. **誰**（名前、メール、ID）。
4. **期待されていた動作**。
5. **実際に発生したこと**（詳細；画像・動画は手戻りを防ぎます）。

### 標準メッセージ

```
ご連絡ありがとうございます。
状況を正確に把握するため、以下の情報をご教示いただけますでしょうか。
・発生した画面またはURL
・発生日時
・対象者のお名前／ID/メールアドレス
・期待されていた動作
・実際に発生した事象（可能でしたら画像・動画もご共有ください）
いただいた情報をもとに速やかに調査を進めさせていただきます。
何卒よろしくお願い申し上げます。
```

---

# 4. クライアントからの質問

最低要件：

1. **明確な文脈** – 質問に根拠または参照があるか？
2. **タスクID**（存在する場合）。
3. **場所** – 質問が関係する画面・フロー。
4. **一貫性** – 提示された文脈で質問が意味を成すか？

### 標準メッセージ

```
ご質問ありがとうございます。
恐れ入りますが、本件の文脈が確認できなかったため、
以下についてご教示いただけますでしょうか。
・本質問に関連するタスクID
・どの画面／どの処理についてのご質問か
・背景または経緯
お手数をおかけしますが、正確に確認するため、
ご協力をお願いいたします。
```

---

# 5. デプロイリクエスト

最低要件：

1. **月曜日または火曜日**を希望日として。
2. **緑環境（customer-preview）でのテスト確認**。
3. **アーキテクチャが決定**：デプロイは日中か夜間か。

### 標準メッセージ

```
ご依頼ありがとうございます。
青い環境への反映に際し、以下の点を確認させてください。
・反映日は「月曜」または「火曜」で問題ございませんか
・緑環境での動作確認は完了しておりますでしょうか。
確認の上、スケジュールを確定させていただきます。
引き続きよろしくお願い申し上げます。
```

---

# 6. 調整リクエスト

考えられるケース：

### (i) 期限が不明

```
ご連絡ありがとうございます。
念のため確認させていただきたいのですが、
本件のご希望の納期（締切日）はいつ頃を想定されておりますでしょうか。
正確なスケジュール調整のため、ご教示いただけますと幸いです。
```

### (ii) 優先度が不明

```
ご依頼ありがとうございます。
本件の優先度につきまして、【優先順位1～3】、または「緊急」など、
ご希望のレベルをご教示いただけますでしょうか。
優先度に応じて社内の対応順序を調整させていただきます。
```

### (iii) 理解の確認

```
ご連絡ありがとうございます。
正確に対応するため、本件の意図およびご要望内容が
合っているか確認させてください。
（※こちらで理解している内容を簡単に箇条書き）
上記に相違がないかご確認いただけますでしょうか。
問題なければ、開発部に共有して進めさせていただきます。
```

### (iv) 方向転換

```
ご連絡ありがとうございます。
念のため確認ですが、
「現在進めている作業を一旦停止し、別の作業を優先する」
というご指示でお間違いございませんでしょうか。
本件はプロジェクト全体に影響するため、ご教示いただけますと幸いです。
ご確認のほどよろしくお願いいたします。
```

---

# 7. クライアントの回答（不完全）

基準：

* リクエストされた内容の一部のみに回答。

### 標準メッセージ

```
ご返信ありがとうございます。
いただいた内容のうち、以下の点についてまだ確認できておりませんため、
追加のご回答をお願いできますでしょうか。
・（※未回答の項目を箇条書き）
お手数をおかけいたしますが、よろしくお願いいたします。
```

---

# 8. ノイズ

### (i) 理解不可能なメッセージ

```
ご連絡ありがとうございます。
恐れ入りますが、いただいた内容の意図を正確に理解できませんでした。
お手数ですが、もう少し詳しくご説明いただけますでしょうか。
正確に対応するため、ご協力をお願いいたします。
よろしくお願い申し上げます。
```

### (ii) 複数のトピックが混在

```
ご連絡ありがとうございます。
本件、複数のテーマが一度に含まれているように見受けられましたため、
正確に確認するためにも、以下のように話題を分けていただけますでしょうか。
（例：①◯◯ ②△△ ③□□）
お手数をおかけいたしますが、ご協力いただけますと幸いです。
よろしくお願い申し上げます。
```

# 9. 新しいカテゴリー

既存のカテゴリーに当てはまらない場合は、アーキテクチャ／Diniz部門に相談してください。

---

# 10. 最高ルール — 「疑わしいときは尋ねる」

カテゴリー、最低要件、クライアントの意図、またはリクエストの一貫性について疑問がある場合：

* **アーキテクチャに進めない**
* **何も仮定しない**
* **説明を求める**

ユニバーサルメッセージ：

```
ご連絡ありがとうございます。
念のため確認させていただきたいのですが、
本件についての意図や背景が明確に読み取れなかったため、
もう少し詳しくご説明いただけますでしょうか。
正確に対応するため、ご協力をお願いいたします。
```
